# PRD: Self-hosted дашборд по метрикам Amplitude (Impala + Postgre)

**Статус:** Draft v1
**Дата:** 2026-06-30
**Проект:** `metrics_audit` (расширение)

---

## 1. Краткое описание

Нужен лёгкий веб-дашборд, который показывает агрегированную таблицу метрик, собранную из двух источников — Impala (`drb`, `external_sources`) и Postgre. Данные собираются на корпоративной машине по расписанию, выгружаются как статический снапшот и публикуются на бесплатном хостинге. Пользователи открывают дашборд с любого устройства, входят по паролю и смотрят данные в виде дашборда + листа детализации (по образцу Qlik Sense). Поверх загруженного снапшота в браузере работает SQL-инструмент для ad-hoc анализа.

---

## 2. Цели и не-цели

### Цели
- Собрать одну общую таблицу из нескольких таблиц Impala и Postgre.
- Автоматическое обновление данных по расписанию (cron-подобно) на корп-машине.
- Веб-интерфейс с двумя режимами: **Дашборд** (визуализации) и **Детализация** (таблица с фильтрами).
- SQL-инструмент в двух местах: (а) админ пишет SQL к реальным БД на корп-машине; (б) любой залогиненный пользователь пишет SQL поверх снапшота прямо в браузере.
- Две роли: **admin** (загрузка данных, настройка SQL и расписания, создание листов) и **viewer** (только просмотр).
- Минимальный стек, бесплатный хостинг, вход с разных устройств по логину/паролю.

### Не-цели (v1)
- Live-запросы к Impala/Postgre из браузера. Браузер на чужом устройстве не в корп-сети — это невозможно. Данные всегда читаются из снапшота.
- Запись/изменение данных в источниках. Дашборд read-only по отношению к БД.
- Полноценная многопользовательская система прав (row-level security, аудит и т.п.).
- Real-time стриминг. Минимальный интервал обновления — десятки минут / часы.

## 3. Пользователи и роли

| Роль | Где работает | Права |
|---|---|---|
| **Admin** | На корп-машине (агент) + веб | Настройка подключений, написание/правка SQL к источникам, запуск и расписание сборки, публикация снапшота, конфигурация листов дашборда. В вебе — всё, что viewer + редактор листов. |
| **Viewer** | Только веб | Вход по паролю, просмотр дашборда и детализации, ad-hoc SQL поверх снапшота (без доступа к самим БД). |

> Роль определяется паролем при входе (два разных пароля → две роли). Это слабое разграничение; см. раздел 8 о рисках.

## 4. Архитектура

Два слабосвязанных компонента: **агент сборки** (корп-машина) и **статический веб-дашборд** (хостинг).

```
┌─────────────────────────── Корпоративная сеть (VPN / corp Wi-Fi) ───────────────────────────┐
│                                                                                              │
│   Корп-машина (всегда онлайн)                                                                 │
│   ┌────────────────────────────────────────────────────────────┐                            │
│   │  Python-агент (planner + collector)                         │                            │
│   │   • расписание (APScheduler / Task Scheduler / cron)        │                            │
│   │   • impyla → Impala  (патч SSL из инструкции)               │                            │
│   │   • psycopg → Postgre                                       │                            │
│   │   • выполняет admin-SQL → собирает общую таблицу (≤50k)     │                            │
│   │   • сериализует снапшот (Parquet + meta.json)               │                            │
│   │   • публикует (git push / API хостинга)                     │                            │
│   └────────────────────────────────────────────────────────────┘                            │
│                                   │ публикация снапшота                                       │
└───────────────────────────────────┼──────────────────────────────────────────────────────────┘
                                     ▼
                      ┌──────────────────────────────┐
                      │ Хостинг статики (бесплатный)  │
                      │  • data/snapshot.parquet      │
                      │  • data/meta.json             │
                      │  • index.html + JS бандл      │
                      └──────────────────────────────┘
                                     │ HTTPS
                ┌────────────────────┴────────────────────┐
                ▼                                          ▼
        Устройство admin                          Устройство viewer
        (вход по паролю)                          (вход по паролю)
        Дашборд + Детализация + SQL поверх снапшота (DuckDB-WASM)
```

**Ключевой принцип:** браузер никогда не подключается к Impala/Postgre. Он читает только заранее собранный снапшот. Всё «общение с БД» происходит исключительно на корп-машине.

## 5. Компонент A — Агент сборки (корп-машина)

### 5.1. Назначение
По расписанию подключается к источникам, выполняет настроенные SQL-запросы, собирает общую таблицу и публикует снапшот.

### 5.2. Подключения
- **Impala:** через `impyla` по инструкции `Impala_connection.md`. Обязателен monkey-patch SSL-контекста (`_patch_thrift_ssl()`) до подключения, иначе `SSLV3_ALERT_HANDSHAKE_FAILURE`. Основной хост `bdas-worker-08`, резервный `bdas-utility-01` — переключение по try/except.
- **Postgre:** через `psycopg[binary]` (psycopg3) или `psycopg2`. Стандартный DSN: host, port, dbname, user, password, sslmode по требованию сервера.
- Креды обоих источников — в `config.yaml`, не коммитятся (в `.gitignore`).

### 5.3. SQL-конфигурация
- Запросы хранятся в файлах `queries/*.sql` (под версионированием, без кредов).
- Конфиг `pipeline.yaml` описывает: какие запросы к какому источнику, как объединяются (UNION / join по ключу) в общую таблицу, имя выходного датасета.
- Базовый агрегирующий запрос — из инструкции (TelecomKz + Aitu whitelist + Loyalty), результат: `platform, metrics, entry_date, cnt`.
- Admin правит SQL локально и перезапускает сборку — это и есть «SQL-инструмент к реальным БД».

### 5.4. Сборка снапшота
1. Выполнить каждый запрос, получить DataFrame (pandas).
2. Объединить в общую таблицу согласно `pipeline.yaml`.
3. Валидация: непустой результат, ожидаемые колонки, число строк в разумных пределах.
4. Записать в `data/snapshot.parquet` (компактно, типизировано) + `data/meta.json`:
   ```json
   {
     "generated_at": "2026-06-30T08:00:00+05:00",
     "row_count": 41230,
     "columns": [{"name": "platform", "type": "string"}, ...],
     "source_versions": {"impala": "...", "postgre": "..."},
     "status": "ok"
   }
   ```
5. **Atomic publish:** писать во временный файл → переименовать, чтобы дашборд не прочитал полу-записанный снапшот.

### 5.5. Публикация
- Вариант по умолчанию: `git add data/ && git commit && git push` в репозиторий хостинга.
- Альтернатива: API/CLI хостинга (Cloudflare Pages, Netlify).
- При неуспешной сборке — старый снапшот остаётся, в `meta.json` пишется `status: stale` + ошибка; новый снапшот не публикуется.

### 5.6. Расписание
- Реализация: **APScheduler** внутри long-running процесса агента, либо системный планировщик (Windows Task Scheduler — учитывая Windows-окружение из инструкции).
- Конфиг расписания (интервал / cron-выражение) — в `pipeline.yaml`, правит admin.
- Логи каждого запуска: время, длительность, число строк, ошибки → `logs/`.
- Поведение при оффлайне/обрыве VPN: повтор N раз с бэкоффом, затем пропуск с записью в лог.

### 5.7. Зависимости агента
```
impyla==0.19.0
thrift==0.16.0
thrift-sasl==0.4.3
pure-sasl>=0.6.2
psycopg[binary]>=3.1     # или psycopg2-binary
pandas
pyarrow                  # parquet
pyyaml
apscheduler              # если планировщик внутри процесса
```

## 6. Компонент B — Веб-дашборд (статический)

### 6.1. Стек (минимальный)
- Чистый статический сайт: один HTML + JS-бандл, без бэкенда.
- **DuckDB-WASM** — выполняет SQL прямо по `snapshot.parquet` в браузере (читает Parquet нативно, держит десятки тыс. строк легко).
- Лёгкая библиотека графиков: например Observable Plot или Chart.js.
- Без тяжёлых фреймворков; допустимо vanilla JS или минимальный Preact/Alpine. Сборка через esbuild/vite.

### 6.2. Экран входа
- При открытии — форма пароля. Совпало с admin-паролем → роль admin, с viewer-паролем → viewer.
- См. раздел 8 — на чистом GitHub Pages это **не защищает данные**; нужна одна из опций защиты.

### 6.3. Лист «Дашборд»
- Набор виджетов (карточки KPI, линии динамики по `entry_date`, разбивка по `platform`/`metrics`).
- Конфигурация виджетов — JSON-описание (admin редактирует в UI, сохраняется в снапшот-репо рядом с данными как `dashboard.json`).
- Viewer видит виджеты, может менять фильтры (период, платформа, метрика), не может менять состав виджетов.

### 6.4. Лист «Детализация»
- Таблица по общей таблице снапшота с сортировкой, фильтрами по колонкам, поиском, экспортом в CSV.
- Пагинация/виртуализация для плавности на десятках тыс. строк.

### 6.5. SQL-инструмент (в браузере)
- Поле ввода SQL → выполняется DuckDB-WASM по снапшоту (таблица доступна как, например, `data`).
- Результат — таблица + возможность построить из неё быстрый график.
- Доступен обеим ролям (это анализ локальных данных, не доступ к источникам).
- Admin дополнительно может «закрепить» SQL-результат как виджет дашборда.

### 6.6. Индикатор свежести
- Везде показывать `generated_at` из `meta.json` («Данные обновлены: ...»).
- Если `status: stale` — заметная плашка.
- Автоперезагрузка снапшота: периодический фоновый запрос `meta.json`; при изменении `generated_at` — мягко обновить данные. Это и есть «дашборд обновляется автоматически» со стороны клиента.

## 7. Поток данных (end-to-end)

1. Срабатывает расписание на корп-машине.
2. Агент подключается к Impala (+ SSL-патч) и Postgre.
3. Выполняются настроенные SQL-запросы, собирается общая таблица.
4. Снапшот (Parquet + meta) пишется атомарно и публикуется на хостинг.
5. Открытые в браузерах дашборды видят новый `meta.json` и подтягивают свежие данные.
6. Пользователь смотрит дашборд/детализацию или гоняет свой SQL по снапшоту.

## 8. Безопасность и приватность (важно)

**Заявленный уровень:** простой пароль на вход, данные «публичны если пароль утёк».

**Проблема:** на чистом **GitHub Pages** снапшот (`snapshot.parquet`, `meta.json`) лежит публичным файлом по прямому URL. Любой, кто знает/угадает путь, скачивает данные **в обход экрана входа** — JS-пароль защищает только UI, не файлы. Фактически данные становятся публичными сразу, не «если пароль утёк». Для корпоративных метрик это, скорее всего, неприемлемо.

**Опции (выбрать одну):**

- **Опция 1 — Cloudflare Pages + Access (рекомендуется, бесплатно).** Серверная авторизация перед отдачей любых файлов. Реальная защита и снапшота, и страницы. Хостинг бесплатный, вход с любых устройств. Меняет только способ публикации в агенте.
- **Опция 2 — GitHub Pages + шифрование снапшота.** Агент шифрует снапшот симметричным ключом; ключ выводится из пароля на клиенте; браузер расшифровывает в памяти. Файл публичен, но бесполезен без пароля. Минимальный стек сохраняется, но управление паролем/ротацией на тебе.
- **Опция 3 — Приватный репозиторий + закрытый хостинг с логином.** Если устроит доступ через аккаунты сервиса, а не «свой» пароль.

> Решение по этой опции — единственный открытый блокер дизайна. Всё остальное от него не зависит.

**Прочее:**
- Креды БД — только на корп-машине, в `config.yaml`, в `.gitignore`. Никогда не попадают в снапшот, бандл или репо.
- Снапшот не должен содержать сырых PII-полей сверх необходимого для дашборда (агрегаты `platform/metrics/entry_date/cnt` безопасны; если добавляются пользовательские поля — ревью на PII).
- HTTPS на хостинге обязателен.

## 9. Технические допущения и ограничения
- Корп-машина всегда онлайн и в корп-сети (VPN/Wi-Fi) на момент сборки.
- Общая таблица ≤ ~50k строк → целиком грузится в браузер без проблем.
- Если данные перерастут лимит браузера — потребуется предв. агрегация/чанкинг (вне v1).
- Windows-окружение агента (из инструкции): использовать `pure-sasl`, не `sasl`.

## 11. Этапы (предложение)

1. **MVP агента:** подключения к обоим БД, один SQL, выгрузка Parquet+meta локально.
2. **MVP веба:** загрузка снапшота, лист «Детализация», индикатор свежести.
3. **Дашборд:** виджеты + фильтры, конфиг `dashboard.json`.
4. **SQL-инструмент в браузере** (DuckDB-WASM).
5. **Расписание + автопубликация + автоперезагрузка** на клиенте.
6. **Авторизация/защита** по выбранной опции из раздела 8.
7. Роли admin/viewer, редактор листов для admin.
